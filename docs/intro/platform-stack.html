agentserver 平台层

agentserver 真正的发明不是任何一个子组件——sandboxproxy、cc-broker、credentialproxy 都只是物化。它真正做对的事是:让 agent 成为 workspace 的一等公民。 可以被命名、被发现、被反向连进、可以互相 peer-proxy。这件事一旦成立, "分布式 coding-agent"才有可能。

五个核心概念

概念是什么解决的问题
workspace 多 agent 共享的隔离域,有一个 UUID + 一个人类可读的别名 多团队/多项目并存的边界
agent workspace 中的命名成员,有 agent_id + display_name + card(能力公告) 算力的可寻址性 + 可发现性
task 跨 agent 的异步 RPC;有 task_id、skill、prompt、生命周期状态 算力的可调用性 + 可追踪性
session chat 类 task 的连续上下文;由 backend(Claude/Codex)持有 跨 task 续接 LLM 状态(humanloop 的基础)
tunnel + peer-proxy 反向 WebSocket(slave→server)+ agent↔agent HTTP 直连(/api/agent/peer/<short_id>/proxy) NAT 穿透 + workspace 内成员互通

为什么 peer-proxy 是关键

没有 peer-proxy,driver 给 slave 派任务时,如果任务带文件,slave 要"取文件" 就得知道 driver 的公网地址——driver 通常在用户笔记本上,根本没有公网地址。 peer-proxy 让 slave 通过 agentserver 反向回连 driver 自己,文件流借道走过去。 这是"workspace 内一等公民"的具象体现。

子组件:都是上面这个发明的物化

agentserver 仓[4]里这些 Dockerfile 不是"功能堆叠", 而是把"agent 是一等公民"这个抽象拆解到工程可实现的边界。每一个解决一个具体的"不是一等公民则会破"的痛点:

子组件不存在它,什么破?
sandboxproxy 同 workspace 多 agent 共用文件系统/进程空间 → 一个 agent 越权改另一个 agent 的状态
credentialproxy API key 暴露给 agent 本体 → agent 一旦被 prompt-inject 就能把 key 外发
llmproxy 所有 agent 写死 api.openai.com / api.anthropic.com → 不能换成自建网关/合规端点
cc-broker Claude Code 多实例没有中转 → 同 host 跑多个 chat 时各自占端口冲突
executor-registry 不同形态 agent(nanoclaw/openclaw/opencode)各自暴露不同 API → 上层无法统一调用
nanoclaw / openclaw / opencode 不同重量级的 coding-agent 形态都要重复实现 sandbox/proxy/registry 的对接
imbridge 外部 IM(聊天)消息没法接入 agent 任务流

接口快照

"agent 是一等公民"这件事在 Go SDK[4]里长成这样 (github.com/agentserver/agentserver/pkg/agentsdk):

type AgentCard struct {
    AgentID     string          // 跨 workspace 唯一
    DisplayName string          // 人类可读的别名
    Status      string          // available / busy / offline
    Card        json.RawMessage // 自定义公告:skills/tools/resources/...
}

type DelegateTaskRequest struct {
    TargetID       string  // 派给谁
    Skill          string  // 路由用的能力名
    Prompt         string  // 任务体
    TimeoutSeconds int
}

type DelegateTaskResponse struct {
    TaskID    string
    SessionID string  // chat backend 的 session,跨 task 续接的钩子
    Status    string  // 初始状态
}

type TaskInfo struct {
    TaskID, Status, SessionID, TargetID string
    Result    json.RawMessage  // 结构化输出
    Output    string           // 人类可读输出
    // ... 加 timing / cost / failure_reason 等
}

注意:SessionID 已经是 agentserver 的一等字段——本栈的 humanloop pause/resume[2] 就是踩在这个字段上立起来的。
