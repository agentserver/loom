## 为什么 unit 测试足够（无需 host-local e2e）

§1.5 的两个 fix 是 **纯进程内的 prompt 拼接 + 控制流改动**：

1. `wrapUserContent` 是字符串处理 helper（escape + truncate）
2. `Plan` / `Route` 的重试与白名单校验是单进程内的循环 + map 查表

没有跨服务调用、没有 transport 改动、没有新的网络/磁盘语义。攻击向量 —— 恶意 LLM 输出闭合标签 / 幻觉 agent id / freeform 文本 —— 在 unit 测试里可以直接构造投递，比 e2e 更直接且更对抗。

对比 §1.3（control plane 跨 driver↔observer↔slave）必须本机灰度跑 prod_test 才能验；§1.5 不属于这一类。

### #20 prompt boundary — `internal/planner/prompts.go`

| 测试 | 钉住的不变量 |
|---|---|
| `TestWrapUserContent_Basic` | helper 给定 tag 与正常文本时按 `<tag>...</tag>` 包裹 |
| `TestWrapUserContent_EscapesClosingTag` | 输入若含 `</tag>`，被 escape 成 `<\/tag>`，不会破坏外层结构 |
| `TestWrapUserContent_TruncatesOversized` | 超过 64 KiB 的 payload 被截断（cap 后追加 truncation marker） |
| `TestWrapUserContent_TruncatesAtRuneBoundary` | 截断点不能切在 UTF-8 多字节 rune 中间（防止 `invalid utf-8`） |
| `TestPlanPrompt_WrapsTaskPromptInBoundary` | `Plan` 拼 system prompt 时把任务 prompt 放到 `<user_task>...</user_task>` 中 |
| `TestPlanPrompt_EscapesMaliciousUserTaskClosingTag` | 用户 prompt 含 `</user_task>` 越权指令，被 escape 后不破坏 LLM 上下文 |
| `TestRoutePrompt_WrapsTaskPromptInBoundary` | `Route` 的 system prompt 同样走 `<user_task>` boundary |
| `TestReducePrompt_WrapsSubOutputsInBoundary` | reducer 把每个子 agent 输出包到 `<node_output id="...">...</node_output>` |
| `TestReducePrompt_EscapesMaliciousNodeOutput` | 子 agent 返回的恶意 closing tag 被 escape，attacker 无法跨节点串改 reduce 决策 |

> spec 里写的是 "8 个 prompts_test.go 测试"，实际实现是 9 个（helper 测试细化为 4 个独立 case）。功能覆盖等价。

### #21 Plan/Route 重试 + 白名单 — `internal/planner/planner.go`

8 个新测试，文件 `internal/planner/planner_test.go`：

| 测试 | 钉住的不变量 |
|---|---|
| `TestPlan_RetriesOnJSONParseError` | Plan 收到非 JSON 输出 → 不立刻返错，进入下一轮重试 |
| `TestPlan_RetriesOnUnknownTargetID` | Plan 收到 target_id 不在 discovered agent 集中 → 拒绝并重试 |
| `TestPlan_SucceedsOnSecondAttempt` | 第二轮 LLM 返回合法 JSON → Plan 成功，不报错 |
| `TestPlan_GivesUpAfter3Attempts` | 3 轮全失败 → 返 `plan rejected after 3 attempts; last error: ...` |
| `TestRoute_RetriesOnJSONParse` | Route 也按相同 3 次循环重试 parse 错误 |
| `TestRoute_RetriesOnUnknownTargetID` | Route 返回未注册 agent → 重试而非沿用 |
| `TestRoute_AcceptsEmptyTargetID` | 空 target_id 是合法 "无合适 agent" 信号，不算错 |
| `TestRoute_NoLongerFallsBackToRawTrim` | 删除 `strings.TrimSpace(out)` 兜底后，LLM freeform 文本（"Sure, here is the answer:"）不再被当 agent id |

实现要点：
- 每轮 attempt 用同一 system prompt 重投 LLM；只 retry 解析层失败，不 retry 业务错误
- `validatePlanAgentIDs` 在 parse 后立刻做白名单 (`map[string]struct{}` of discovered AgentID)
- Route 同样做 whitelist 校验；空字符串显式当作"无候选"返回 (`""`, nil)，不视为失败

### 已知失败（不在 §1.5 修复范围 — master 冻结）

```
FAIL  github.com/yourorg/multi-agent/internal/orchestrator   1.647s
  --- FAIL: TestFanout_EmitsPlanningCompleted
  --- FAIL: TestFanout_MCPValidationReplanRejectsDisallowedContractTarget
```

两个失败都是 §1.5 紧化语义触发的下游 orchestrator 测试对旧行为的依赖：

1. **`TestFanout_EmitsPlanningCompleted`** — 测试用 `plan_chain` 桩（引用 `agent-a` + `agent-b`），SDK 只注册 `agent-a`。§1.5 的 `validatePlanAgentIDs` 现在拒绝未知 `target_id`，Plan() 直接报 `unknown agent_id "agent-b"`，orchestrator 永不进 `planning_completed` 路径。**这是测试桩与新校验的对齐问题，新校验本身是 Bug #21 的核心修复，不能回退**。
2. **`TestFanout_MCPValidationReplanRejectsDisallowedContractTarget`** — bash planner 桩在 `round >= 2` 时输出字面 `"REDUCED"`。§1.5 重试 3 次后报 `plan rejected after 3 attempts`，老的 fail-fast 文案 `invalid mcp validation replan: target agent-b is not allowed` 不再匹配。**桩需要在每个 attempt 都输出合法 JSON，或测试断言要松绑**。

两个修复都是 **orchestrator 测试文件 + testdata/fake-planner.sh** 的对齐，**不需要改 §1.5 产代码**。已 [filed as issue #16](https://github.com/agentserver/loom/issues/16)，由 orchestrator 后续 PR 处理（master 路径冻结，本 PR 不动）。
